## Auditoría pre-lanzamiento (2026-03-22)

**Why:** Auditoría completa antes del go-live en zen-trader.com

### Hallazgos críticos encontrados

1. **Import CSV sin plan gating server-side** — `app/api/trades/import/route.ts` no llama a `getUserPlan()`. Cualquier usuario Free/Starter puede importar trades via curl aunque el UI bloquee el acceso.

2. **affiliate_conversions sin UNIQUE en transaction_id** — Si el webhook de Wompi se llama dos veces con el mismo transaction_id (retry de Wompi), se insertan dos registros de comisión duplicados. No hay `ON CONFLICT` ni unique constraint.

3. **increment_affiliate_uses RPC sin SECURITY DEFINER** — La función PostgreSQL corre con permisos del invocador (SECURITY INVOKER por defecto). Llamada via `supabaseAdmin` (service_role) está bien, pero el patrón es frágil si se expone en otro contexto.

4. **Migración 006 diverge del schema real** — El archivo `supabase/006_subscriptions.sql` tiene `lemon_squeezy_subscription_id TEXT NOT NULL UNIQUE` pero el schema real (ver `types/database.types.ts`) usa `processor_subscription_id`. No hay archivo SQL de migración que documente esta transición. Riesgo: re-ejecutar migraciones en producción o staging fallaría.

5. **Vercel Cron sin autenticación nativa** — Vercel llama los crons sin header Authorization (a menos que se configure explícitamente). Los crons verifican `Bearer ${CRON_SECRET}` en el código, pero en Vercel Pro/Enterprise el cron invoca la URL directamente. Verificar que `CRON_SECRET` está en Vercel env y que Vercel envía el header.

### Hallazgos WARNING (no críticos)

- `billing/cancel` usa `supabase` (user client) para actualizar subscriptions. El RLS permite solo SELECT a usuarios autenticados — si la política en producción tiene `FOR SELECT ONLY`, el UPDATE fallará silenciosamente. Verificar que la política de subscriptions cubre UPDATE del propio usuario.
- `newsletter_subscribers` permite inserts anónimos — correcto para el formulario público, pero no hay rate limiting. Un bot podría llenar la tabla.
- `app/api/trades/import` usa `any` dos veces (líneas 81 y 125) — viola TypeScript strict del proyecto.
- Soporte de herramienta interna (`/api/agents/support`) autenticada solo por `SUPPORT_AGENT_SECRET` — correcto para n8n, pero asegurarse que la var está en producción.

### Patrones que funcionan bien (para referencia futura)

- Auth pattern en API routes: `const { data: { user } } = await supabase.auth.getUser(); if (!user) return 401`
- Checkout con pending_checkouts: el mapeo linkId→usuario está bien implementado
- validateWebhookChecksum: correcto, rechaza si WOMPI_EVENTS_KEY no está configurado (falla segura)
- Validación de monto en webhook: verifica `amount_in_cents >= expected` antes de activar plan
- supabaseAdmin: solo en API routes y crons, nunca en componentes cliente
- Cron auth: `Bearer ${CRON_SECRET}` con fallo si `!cronSecret`
- Admin panel: doble check (Server Component + API route) con `isAdmin(email)` basado en `ADMIN_EMAILS` env var
- RLS en subscriptions: SELECT a usuarios autenticados, ALL a service_role

**How to apply:** En auditorías futuras, verificar primero import/export routes (típicamente olvidan plan gating). Siempre verificar idempotencia en tablas de comisiones/pagos.
